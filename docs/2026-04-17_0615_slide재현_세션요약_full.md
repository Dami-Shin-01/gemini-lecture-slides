## 3.2 Chapter (`slide--chapter`)

- 배경: `var(--color-bg)` (오프화이트)
- 우측 뒤에 거대한 챕터 번호(`chapter-number-bg`, clamp 160~300px, opacity 0.07, letter-spacing -0.05em)
- 좌측 65% max-width 본문: `chapter-tag` (대문자 라벨, 챕터색) / `chapter-title` (h1) / `chapter-desc` (body, 480px max)
- 하단 전폭 5px `chapter-accent-line` 챕터색

## 3.3 Content (`slide--content`)

- 좌측에 6px `slide-accent-bar` (챕터색) — padding-left에 6px 추가
- `content-header`: `content-chapter-label` (대문자) + `content-title` + 선택 `content-subtitle`
- `content-title::after`: 48px × 3px 챕터색 pill (타이틀 강조 언더바)
- 본문: `bullet-list` (flex column, gap md, 중앙정렬)

- 선택 `callout`: 챕터 light 배경 + 4px 좌변 챕터색, `strong`은 챕터색

## 3.4 Compare (`slide--compare`)

- `compare-grid`: `grid-template-columns: 1fr auto 1fr`

- Before(검색): `"보고서 작성법" 입력` → 관련 글 10개 링크, 내가 조합해야 / 키워드 정확·직접 가공·여러 페이지
- After(AI): `"우리 팀 월간 보고서 초안 써줘"` → 바로 쓸 수 있는 문서 / 말하듯이 OK·내 상황 가공·이어서 수정
- notes: 좌우 비교로 시각화. 같은 니즈를 두 가지 방식으로 해결하는 과정을 직접 시연하면 가장 강력합니다.

**[6] (1/content) AI는 '대화형 도구'다** — 한 번 묻고 끝이 아니라, 주고받으며 완성된다
- bullets: [1] 1차 요청: "회의 안건 목록 만들어줘" / [2] 2차 수정: "좀 더 구체적으로, 마케팅팀 월간 회의 기준으로" / [3] 3차 보완: "맨 위에 전월 실적 검토 항목 추가해줘"
- callout: **핵심:** 3번 대화하면 혼자 30분 걸릴 일이 5분으로 줄어든다
- notes: AI를 처음 접하는 분들의 가장 큰 실수는 '첫 질문 하나로 완벽한 결과를 기대하는 것'. 대화를 이어가는 것이 자연스럽다는 점을 강조하세요.

**[7] (1/content) AI가 잘하는 일 vs 주의할 일**
- bullets: [O] 초안 작성, 요약, 번역, 아이디어 나열, 형식 변환 / [O] 반복적인 문서 패턴 생성 / [!] 최신 뉴스·오늘 날짜 기반 정보는 학습 시점 한계 / [!] 숫자 계산이나 사실 확인은 반드시 검토
- callout: **원칙:** AI 결과물은 '출발점'이지 '최종본'이 아니다
- notes: AI의 한계를 솔직하게 얘기하는 것이 신뢰를 높입니다. '맹신하지 말고 검토하라'는 메시지를 자연스럽게 심어주세요.

**[8] (1/content) Gemini의 3가지 핵심 강점** — 왜 사무직 실무자에게 적합한가
- bullets: [1] 구글 생태계 통합 — Gmail, 문서, 스프레드시트, 슬라이드와 직접 연결 / [2] 멀티모달 — 텍스트뿐 아니라 이미지, PDF 파일을 함께 분석 / [3] 최신 정보 접근 — 구글 검색과 연계되어 실시간 정보 반영
- callout: **핵심:** 구글 워크스페이스를 쓴다면 Gemini가 가장 자연스럽게 연결된다
- notes: 구글 워크스페이스를 회사에서 쓰는 수강생이라면 Gemini가 특히 강력. Gmail에서 바로 답장 초안을 잡아주는 기능 등 실제 화면을 보여주면 반응이 좋습니다.

**[9] (1/content) Gemini, 어디서 어떻게 접근하나**
- bullets: [→] 웹: gemini.google.com — 브라우저에서 바로 사용 / [→] 모바일: Gemini 앱 설치 후 구글 계정 로그인 / [→] Gmail/Docs 내장: '도움받기' 버튼 / [→] 무료 플랜으로 핵심 기능 대부분 사용 가능
- notes: 수강생들이 지금 바로 화면을 열어볼 수 있도록 유도하세요. 회사 계정과 개인 계정의 차이(보안 정책)도 간략히 언급합니다.

**[10] (1/content) 실무에서 Gemini를 쓰는 5가지 장면**
- bullets: [1] 이메일 답장 초안: "이 메일에 정중하게 거절하는 답장 써줘" / [2] 회의 준비: "다음 주 예산 검토 회의 안건 리스트 만들어줘" / [3] 문서 요약: PDF 붙여넣고 "핵심 3줄로 요약해줘" / [4] 아이디어 발산: "신제품 홍보 문구 10가지 제안해줘" / [5] 형식 변환: "이 데이터를 표 형식으로 바꿔줘"
- notes: 수강생의 '나도 쓸 수 있겠다'는 확신을 만드는 슬라이드. 참석자 직종에 맞는 예시를 1-2개 맞춤형으로 언급하면 몰입도가 올라갑니다.

**[11] (1/diagram) 3단계 실무 적용 로드맵**
- flow: 1주(매일 하나씩 — 이메일 초안, 요약 한 가지만 AI에 맡긴다) → 1달(프롬프트 모음 — 자주 쓰는 요청을 저장해두고 재활용) → 3달(팀 공유 — 잘 만든 프롬프트를 팀원과 나눈다)
- notes: 로드맵을 3단계 화살표 다이어그램으로 시각화. '3개월 후 팀 내 AI 리더가 될 수 있다'는 긍정적 비전을 제시하세요.

- callout: **기억하세요:** 결과가 완벽하지 않아도 괜찮습니다 — 써봤다는 것 자체가 성공입니다
- notes: 챕터 1 마무리. 수강생이 강의장을 나가기 전에 '내일 당장 할 것 하나'를 머릿속에 심어주세요.

## 5.3 Chapter 2 — 프롬프트 엔지니어링 (레드)

**[13] (2/chapter) 프롬프트 엔지니어링** — 원하는 답을 끌어내는 프롬프트 구조를 익힌다
- notes: 챕터 2 시작. '프롬프트 = AI에게 보내는 메시지'라고 짧게 정의해주세요.

**[14] (2/diagram) 대화 캔버스: PTCF 프레임워크**
- flow: P(Persona — AI에게 역할 지정) → T(Task — 구체적인 작업 요청) → C(Context — 내 상황과 배경) → F(Format — 결과물 형태 지정)
- notes: PTCF를 시각적으로 표현. 4가지를 한 번에 쓰려고 부담 갖지 말고, P와 T만 있어도 충분히 좋은 출발점이 된다고 안내하세요.

**[15] (2/content) P: 역할(Persona) 지정하기** — '너는 ~야' 한마디가 결과의 톤과 깊이를 바꾼다
- bullets: [X] 역할 없음: "보고서 써줘" → 일반적이고 평범한 결과 / [O] "너는 10년 경력 마케팅 기획자야. 보고서 써줘" → 전문적 시각 반영 / [O] "너는 인사팀 담당자야. 신입 직원 온보딩 안내문 작성해줘" / [O] "너는 꼼꼼한 편집자야. 이 문장의 오류와 어색한 부분 찾아줘"
- notes: Persona 지정의 효과를 직접 시연하면 설득력이 높아집니다. 같은 요청에 역할 붙이기 전/후를 비교해 보여주세요.

**[16] (2/content) T+C: 작업(Task)과 맥락(Context)**
- bullets: [T] 작업: "주간 업무 보고서 초안을 작성해줘" "회의 안건 목록을 만들어줘" / [T] 구체적 동사가 핵심: "써줘"보다 "요약해줘" "비교해줘" "검토해줘" / [C] 맥락: "우리 회사는 30명 규모 제조업이고, 안전 교육 계획이 필요해" / [+] 결합: "영업팀 주간 보고용으로, 항목별로 나눠서, 각 항목 2줄씩"
- notes: Task를 명확히 하는 것이 가장 중요합니다. '써줘'보다 '초안을 작성해줘' '목록을 만들어줘'처럼 구체적인 동사를 쓰라고 안내하세요.

**[17] (2/content) PTCF 통합 예시** — 요소를 하나씩 추가할수록 결과물이 내 것처럼 바뀐다
- bullets: [0] 기본: "회의록 써줘" / [P] + 역할: "너는 꼼꼼한 회사 비서야" / [T] + 작업: "신제품 론칭 회의의 회의록을 작성해줘" / [C] + 맥락: "마케팅팀 5명 참석, 결정사항 위주로" / [F] + 형식: "결정사항/액션아이템/담당자를 구분해서 표로"
- notes: 누적 빌드업 형태가 핵심. 처음에 '회의록 써줘' 하나만 보여주고, 요소가 하나씩 붙는 과정을 보여주세요.

**[18] (2/divider) Before / After 프롬프트 비교**
- notes: Before/After 비교 섹션으로 전환. 2-3초 후 넘어갑니다.

**[19] (2/compare) 사례 1: 이메일 작성**
- Before: "거래처 이메일 써줘" → 수신자·목적 불분명 → 어디에도 못 쓰는 일반적 이메일
- After: "영업팀 담당자야. 납기 지연 사과 + 2주 연장 요청 이메일. 15년 거래처 김 부장님, 공식적이지만 친근한 톤, 3문단 이내" → 바로 발송 가능한 완성도 높은 이메일 초안
- notes: Before/After를 좌우로 나눠 실제 Gemini 결과물 스크린샷을 보여주면 가장 설득력이 높습니다.

**[20] (2/compare) 사례 2: 문서 요약**
- Before: "이 보고서 요약해줘" → 내용 전체를 고르게 압축, 핵심이 묻힘
- After: "경영진 의사결정에 필요한 핵심 수치와 결론만 3줄로 뽑아줘" → 회의에 바로 사용 가능한 요약본
- notes: 요약은 수강생들이 가장 즉시 써볼 수 있는 기능. '지금 당장 길고 읽기 싫은 보고서가 있으면 실습에서 써보세요'라고 연결하세요.

**[21] (2/compare) 사례 3: 아이디어 요청**
- Before: "아이디어 줘" → 무엇에 대한 건지 불분명 → 엉뚱한 결과
- After: "예산 50만 원 이하, 사무직 30명 대상, 사무실 내 진행 가능한 분기 팀빌딩 행사 아이디어 5가지" → 실제 검토 가능한 구체적 제안 5가지
- notes: '제약 조건을 알려줄수록 더 좋은 결과가 나온다'는 역발상적 인사이트가 핵심입니다.

**[22] (2/content) 좋은 프롬프트의 공통점**
- bullets: [1] 목적이 명확하다 — 무엇을 만들거나 해결하려는지 분명히 말한다 / [2] 맥락이 있다 — 내 상황, 독자, 배경을 알려준다 / [3] 형식을 지정한다 — 길이, 구조, 말투를 구체적으로 요청한다 / [4] 제약을 숨기지 않는다 — 조건이 많을수록 결과가 더 현실적이다
- callout: **나쁜 프롬프트의 공통점은 딱 하나:** 짧고 막연하다
- notes: 이 슬라이드는 Before/After 클립의 결론이자 실습의 기준점. '내가 가장 놓치고 있는 것 하나를 골라보세요'라고 자기 점검을 유도하세요.

**[23] (2/practice) 프롬프트 반복 개선 실습**
- goal: 같은 목표를 3번 다르게 요청하며 프롬프트 개선 근육을 키운다
- steps: 1) 라운드 1: 머릿속에 있는 대로 프롬프트 입력 / 2) 라운드 2: PTCF 중 빠진 요소를 추가해서 다시 요청 / 3) 라운드 3: 결과를 보고 "이것만 바꿔줘"로 다듬기
- tip: 3개 결과를 나란히 비교하면 어떤 요소가 차이를 만들었는지 보인다
- notes: 실습 시작 전 타이머를 보여주세요. 라운드 1에서 '부족한 게 당연하다, 그게 정상이다'라고 안심시켜야 수강생들이 과감하게 시작합니다.

**[24] (2/practice) 실습 과제 선택**
- steps: A) 다음 주 팀 회의 안건 리스트 만들기 / B) 협력사에게 보낼 일정 변경 안내 이메일 초안 작성 / C) 최근 읽은 긴 문서나 뉴스를 요약하고 핵심 시사점 3가지 뽑기
- tip(추천): 내일 당장 해야 하는 실제 업무 과제를 직접 가져와도 됩니다
- notes: 과제 선택지를 미리 제시하면 '무엇을 해야 하지?'로 고민하는 것을 방지합니다. 옆 사람과 서로 다른 과제를 선택하도록 유도하세요.

**[25] (2/content) 막힐 때 참고하는 체크리스트**
- bullets: [?] 역할을 지정했나? → "너는 ~야" 추가해보기 / [?] 내 상황을 충분히 설명했나? → 회사 규모, 업종, 목적 추가 / [?] 원하는 형식을 말했나? → 길이, 구조, 말투 지정 / [?] 어느 부분이 마음에 안 드나? → "~는 빼고" "~를 더 강조해서"
- notes: 이 슬라이드는 실습 중에 화면에 계속 띄워두면 좋습니다. 수강생들이 강사에게 물어보기 전에 자가 진단할 수 있는 도구입니다.

# 6. JavaScript 동작 명세

## 6.1 전체 구조

IIFE(`(function(){'use strict'; ... })();`) 내부에서 동작. DOM 쿼리·상태·이벤트가 모두 이 스코프에서 관리.

```js
const slides = Array.from(document.querySelectorAll('.slide'));
const total  = slides.length;
let current = 0;
let isAnimating = false;
let notesOpen = false;
let gotoOpen  = false;
```

## 6.2 공개 동작

- `init()`: 전체 슬라이드 수를 카운터에 세팅, `goTo(0, false)`로 첫 슬라이드 활성, 이벤트 바인딩, 5초 후 nav-hint fade out.
- `goTo(index, animate=true)`: 인덱스 검증 + 애니메이션 잠금(480ms) + 각 슬라이드 class 갱신 + progress/counter/notes 업데이트.
- `next()/prev()`: `goTo(current ± 1)`.
- `updateProgress()`: `current / (total-1) * 100%` 폭 + 현재 슬라이드에서 `--ch-color` 값을 읽어 background 전환.
- `updateCounter()`: "1 / 44" 갱신 + 2.5초 표시 타이머.
- `updateNotes()`: 패널 텍스트를 `slides[current].dataset.notes`로 교체.
- `toggleNotes()`: `.is-open` 토글, `aria-hidden` 동기.
- `openGoto() / closeGoto() / submitGoto()`: max=total 설정, Enter 확정 / Esc 닫기.
- `triggerAnimation(slide)`: 슬라이드 전환 시 모든 슬라이드의 `bullet-item / flow-step`에서 `is-visible` 제거 → 현재 슬라이드만 80ms 간격으로 순차 추가 + 시작 딜레이 300ms.

## 6.3 이벤트 바인딩

```js
document.addEventListener('keydown', (e) => {
  if (gotoOpen) { /* Escape=close, Enter=submit */ return; }
  switch (e.key) {
    case 'ArrowRight': case 'ArrowDown': case ' ': e.preventDefault(); next(); break;
    case 'ArrowLeft':  case 'ArrowUp':   e.preventDefault(); prev(); break;
    case 'n': case 'N': toggleNotes(); break;
    case 'g': case 'G': openGoto(); break;
    case 'Escape': if (notesOpen) toggleNotes(); break;
    case 'Home': goTo(0); break;
    case 'End':  goTo(total - 1); break;
  }
});

// 터치: 좌우 스와이프 > 50px && 수평성분 우세
// 마우스: 좌측 클릭=prev, 우측 클릭=next (단, notes-panel/close/goto 영역은 예외)
// mousemove: 슬라이드 카운터 재노출
```

## 6.4 주의점

- `triggerAnimation`은 전역 순회 후 현재 슬라이드만 세팅하므로, 슬라이드 수가 많아져도 상태 누수 없음.
- `goTo`가 애니메이션 중 호출되면 무시됨(480ms 동안). 연타를 막지만 빠른 Key-spam에는 프레임 드롭 느낌이 날 수 있음.
- 모든 클릭이 기본 navigation으로 처리되므로, 슬라이드 내부에 `<a>`나 링크를 추가하려면 `e.target.closest('.slide a')` 등 예외 처리를 추가해야 함.

# 9. 재현 절차 (완전 재현용)

## 9.1 가장 빠른 재현: 콘텐츠만 교체

1. `index.html`을 복사.
2. `:root` 컬러 토큰(`--color-ch1~4`)을 원하는 테마로 교체.
3. `<title>`, cover-title/subtitle/meta 수정.
4. 각 `<section>` 내부 `h1`/`h2`/`p`/`bullet-text`/`data-notes`만 교체.
5. 챕터 수가 4개 이하면 `data-chapter` 값만 줄이고, 5개 이상이면 `--color-ch5` 토큰 추가 + `.slide[data-chapter="5"]` 셀렉터 추가.
6. 브라우저에서 로컬 실행 (`file:///` 또는 `python -m http.server`).

## 9.2 구조 교체까지 포함한 재현

1. **콘텐츠 설계 선행**: 슬라이드별 `{chapter, type, title, subtitle?, body, notes}`를 JSON으로 먼저 작성.
2. **슬라이드 타입 검토**: cover / chapter / content / compare / diagram / practice / divider / ending 중 적합한 것 선택.
   - 원칙 → content / callout
   - 비교 → compare
   - 프로세스·단계 → diagram
   - 할 일·과제 → practice
   - 섹션 전환 → divider
3. **HTML 생성**: 각 슬라이드를 타입별 템플릿에 채워 넣음.
4. **노트 정리**: `data-notes`는 발표자에게 말을 거는 톤으로, 1~2문장.
5. **검증**: 브라우저에서 키보드 순회 1회 + N키로 전 슬라이드 노트 확인 + G키로 랜덤 점프.

# 10. 재현 시 주의사항

- **챕터 컬러 일관성**: `data-chapter`를 바꿔도 `:root`에 정의된 `--color-chN` 토큰이 없으면 컬러가 깨진다. 토큰 추가가 선행.
- **아이콘**: 본 슬라이드의 bullet-icon은 아이콘 폰트가 아닌 **단일 글자(1, 2, O, X, !, ?, →, R, 요 등)**로 구성. 한글 한 글자도 가능. 재현 시 혼용 주의.
- **compare 변형**: [40] 슬라이드는 bad/good 라벨 규칙을 벗어나 인라인 style로 좌측에 챕터색 적용. 단순 복붙 시 주의.
- **최종 정리 슬라이드([43])**: bullet-icon 각각 인라인 style로 챕터별 컬러를 다르게 지정. "전체 정리" 라벨도 인라인 텍스트.
- **챕터 1 구조 특이**: chapter 슬라이드([4])가 cover 바로 뒤가 아니라 diagram([2]) + content([3])이 먼저 나온 뒤에 등장. "오늘의 시간표 → 학습 목표 → Chapter 1 시작" 흐름. 다른 강의로 재현할 때 같은 구조를 유지할지 여부 검토 필요.
- **Chapter 1에만 사전 슬라이드**: [2], [3]은 Ch1 diagram/content로 표시되어 있으나 성격은 "전체 소개". 재현 시 이 구조를 유지하면 진행도 바가 Ch1 블루로 초반부터 채워진다.
- **`data-chapter="0"`의 Ending**: 끝 슬라이드는 Ch0(블랙)로 되어 있어 진행도 바도 블랙으로 돌아감.
- **로컬 CDN 대체**: 폐쇄망/오프라인 시연 환경에서는 Pretendard 웹폰트 CDN 링크를 로컬 `.woff2`로 치환 필요.
